dbt + AWS Athena
Detailed Architecture

A production-grade data transformation pipeline using dbt Core as the transformation layer, Amazon Athena as the serverless SQL engine, S3 as the data lake, and AWS Glue as the metadata catalog — built for scale, reproducibility, and team collaboration.

01

End-to-End Architecture Flow

How every component connects
🪣
Amazon S3
Data Lake / Storage

Amazon S3 is the foundation of the entire architecture. It acts as the central object store for all data — raw ingested files, dbt-transformed outputs, Athena query results, and dbt artifacts like manifest.json.

Data is organized into prefixed "zones": raw/, staging/, intermediate/, marts/, and athena-results/. Files are stored as Parquet or ORC — columnar formats that dramatically reduce Athena query costs by minimizing data scanned.

S3's durability (99.999999999%) and virtually unlimited scale make it the perfect foundation. You only pay for what you store and query — no database instances, no idle compute costs.

s3://bucket/raw/ s3://bucket/staging/ s3://bucket/marts/ s3://bucket/athena-results/ Parquet ORC Snappy compression
📋
AWS Glue
Data Catalog + Crawlers

AWS Glue Data Catalog is the metadata backbone. It stores table definitions, schemas, partition information, and data locations. Athena uses it as its metastore — every table Athena queries must be registered in the Glue Catalog.

Glue Crawlers automatically scan S3 paths, infer schemas from Parquet/CSV/JSON files, and register or update table definitions. You can schedule crawlers to run after ingestion pipelines complete, keeping the catalog fresh without manual DDL.

For dbt, each model materialization (table or view) either creates a new Glue table entry or updates an existing one. The dbt-athena-community adapter handles this automatically by issuing CREATE TABLE AS SELECT (CTAS) statements which write data to S3 and register the table in Glue simultaneously.

Glue Crawler Glue Database Glue Table Partition Registry Schema Evolution
🔍
Amazon Athena
Serverless SQL Engine

Athena is the serverless SQL query engine that sits over S3. It reads data directly from S3 using the schema from Glue Catalog, executes Presto/Trino-compatible SQL, and writes results to a dedicated S3 results bucket. There are no servers to provision, patch, or scale — Athena auto-scales to handle parallel queries.

Workgroups are critical for production: they isolate different teams' queries, enforce per-query data scan limits (cost control), set the results S3 location, and enable per-workgroup CloudWatch metrics. Create separate workgroups for dbt runs, ad-hoc analysis, and dashboards.

dbt connects to Athena via the dbt-athena-community Python adapter, which uses PyAthena under the hood. dbt compiles SQL models and submits them as Athena queries. Tables are materialized as CTAS or INSERT INTO operations, views as Athena views.

dbt-athena-community PyAthena Workgroups CTAS INSERT INTO Presto SQL Cost per TB scanned
🔧
dbt Core
Transformation Layer

dbt (data build tool) is the transformation engine. It provides a framework for writing SQL transformations as modular, version-controlled files. Each model is a .sql file containing a SELECT statement — dbt handles the DDL (CREATE TABLE AS, CREATE VIEW) based on the configured materialization.

The dbt DAG (Directed Acyclic Graph) is automatically inferred from ref() and source() macros in your SQL. dbt resolves dependencies and runs models in the correct order, with optional parallelism. Every model can have associated schema tests (not_null, unique, accepted_values, relationships) and singular tests.

The profiles.yml contains the Athena connection: region, S3 results location, workgroup, and database. No credentials are hardcoded — the AWS SDK credential chain automatically picks up the execution environment's IAM role.

models/ tests/ macros/ seeds/ snapshots/ dbt docs generate ref() / source() DAG execution
🐳
ECS / CodeBuild / MWAA
Compute Host

The compute host is where dbt actually runs. Three common options exist: Amazon ECS (Fargate) runs dbt in a Docker container with a Task Role — ideal for scheduled pipelines triggered by EventBridge. AWS CodeBuild is perfect for CI/CD-triggered runs on pull requests. Amazon MWAA (Managed Airflow) is best when dbt is one operator in a larger workflow with upstream ingestion steps.

In all cases, the compute host assumes an IAM Task Role (or Instance Profile / execution role). The AWS SDK running inside dbt's Python process automatically discovers this role via the instance metadata service (IMDS) — no key injection, no environment variable credentials needed.

ECS Fargate AWS CodeBuild Amazon MWAA EC2 + Instance Profile IMDS v2 Task Role
🛡️
IAM Role
Identity & Permissions

The IAM Execution Role is the identity attached to the compute host. It grants least-privilege access to every AWS service the dbt process needs: Athena query execution, S3 read/write on specific prefixes, and Glue Catalog reads for schema discovery and writes for table creation/updates.

The role's trust policy specifies who can assume it — for ECS tasks this is ecs-tasks.amazonaws.com, for EC2 it's ec2.amazonaws.com, for GitHub Actions it's the OIDC provider. The permission policy uses resource-level scoping — e.g. S3 access is scoped to specific bucket prefixes, not *.

Least privilege Trust policy Resource-level scoping STS AssumeRole OIDC for CI/CD
⚙️
CI/CD Pipeline
GitHub Actions / CodePipeline

The CI/CD pipeline ties together development workflow and production runs. On every pull request, a pipeline runs dbt build --select state:modified+ using slim CI — only running models and tests that changed. This gives fast feedback without running the entire project.

GitHub Actions connects to AWS via OIDC federation — the workflow exchanges a GitHub-signed JWT for temporary AWS credentials via STS. No AWS access keys are ever stored in GitHub secrets. On merge to main, a production run is triggered in ECS/CodeBuild that runs the full dbt project and uploads the resulting artifacts to S3.

Medallion Data Architecture Layers

RAW → BRONZE → SILVER → GOLD → PLATINUM
ℹ️ Why Medallion Architecture?
The medallion (multi-hop) architecture progressively refines data quality at each layer. Raw data is never modified — all transformations are incremental and auditable. Each layer is independently queryable, enabling both operational and analytical use cases.
RAW / Source
S3 Location s3://bucket/raw/{source}/{table}/

Exact copies of source data as delivered by ingestion tools. Never transformed. Used as the audit trail. Formats: JSON, CSV, Avro, Parquet.
source: raw_db dbt source() Freshness checks
BRONZE / Staging
Prefix s3://bucket/staging/

1:1 with source tables. Casts data types, renames columns to conventions, deduplicates, and applies basic filtering. One staging model per source table.
stg_orders.sql stg_customers.sql stg_products.sql
SILVER / Intermediate
Prefix s3://bucket/intermediate/

Joins multiple staging models together, applies business logic, enriches with reference data. Not exposed to end users — stepping stones to marts.
int_orders_enriched.sql int_customer_segments.sql int_product_revenue.sql
GOLD / Marts
Prefix s3://bucket/marts/

Business-facing dimensional models and fact tables. Optimized for dashboards, reporting, and analytics. Schema follows star or snowflake conventions.
fct_orders.sql dim_customers.sql dim_products.sql fct_revenue.sql
PLATINUM / Metrics
Prefix s3://bucket/metrics/

Pre-aggregated metrics, KPI summaries, and ML feature tables. These are the highest-trust, most stable tables — versioned and tested exhaustively.
mrt_daily_revenue.sql mrt_cohort_analysis.sql features_clv.sql
03

dbt Execution Runtime

What happens when dbt run executes
PARSE & COMPILE
dbt reads all .sql model files, Jinja macros, and schema.yml definitions. It resolves all ref() and source() calls, substituting actual Athena database and table names. The result is compiled SQL stored in target/compiled/. dbt also builds the DAG — the dependency graph that determines execution order.
IAM CREDENTIAL RESOLUTION
The AWS SDK (boto3/PyAthena) starts its credential chain: it first checks environment variables (AWS_ACCESS_KEY_ID — none set), then checks the IMDS endpoint for the ECS task's IAM role. It retrieves temporary credentials (access key + secret + session token) with a short expiry. These rotate automatically. dbt never sees or stores these credentials directly.
CONNECTION TEST
dbt-athena-community establishes a connection by making a lightweight Athena API call (list_databases) to verify credentials and network connectivity. If this fails, dbt exits early with a connection error — no models run. This early check prevents partial runs from creating incomplete data states.
DAG EXECUTION (PARALLEL)
dbt traverses the DAG topologically. Models with no remaining dependencies are eligible to run. With --threads 4, up to 4 models run simultaneously — each submitting a separate Athena query. The adapter polls the Athena API for query completion status. CTAS materializations write results directly to S3 as Parquet, register in Glue, and replace any previous version.
INCREMENTAL MODELS
For models materialized as incremental, dbt checks if the table already exists. On first run it performs a full load. On subsequent runs it uses the is_incremental() macro to filter only new/changed records, then issues an INSERT INTO or a merge strategy. With Athena + S3, incremental models append new Parquet files to the S3 prefix partition.
TESTS & FRESHNESS
After models complete, dbt test runs all schema tests (not_null, unique, relationships, accepted_values) as SELECT queries against Athena. Tests that return rows are failures. Source freshness checks query the loaded_at_field timestamp to verify ingestion is recent. Results are written to target/run_results.json.
ARTIFACTS UPLOAD & OBSERVABILITY
Upon completion, the pipeline uploads manifest.json, run_results.json, and catalog.json to an S3 artifacts path. These power dbt docs, enable slim CI (state comparison), and feed observability tools. CloudWatch logs capture all stdout. Athena CloudWatch metrics show query execution times and data scanned per workgroup.

    o.order_id,

Security & IAM Architecture

Zero static credentials — full role-based access
✅ Core Principle
Never use long-term AWS credentials. The AWS SDK credential chain resolves authentication in order: environment variables → shared credentials file → IAM Instance Profile/Task Role → Container credentials. In production, only the Task Role / Instance Profile path is active. This means rotating, auditable, automatically expiring temporary credentials — no secrets to leak.
🛡️ IAM Role — Permissions Policy
  • athena:StartQueryExecution — submit queries
  • athena:GetQueryExecution, GetQueryResults — poll and fetch results
  • athena:StopQueryExecution — cancel hung queries
  • s3:GetObject, PutObject — scoped to bucket/dbt/* and results/*
  • s3:ListBucket — required for CTAS to check existing objects
  • glue:GetDatabase, GetTable — read schema metadata
  • glue:CreateTable, UpdateTable, BatchCreatePartition — register dbt output tables
🐳 ECS Task Trust Policy
  • Principal: ecs-tasks.amazonaws.com
  • Action: sts:AssumeRole
  • ECS injects credentials via IMDS endpoint into the container
  • Credentials expire and rotate every 1 hour automatically
  • CloudTrail logs every AssumeRole event with task metadata
⚙️ GitHub Actions OIDC (CI/CD)
  • Configure AWS as OIDC provider in IAM using GitHub's discovery URL
  • Trust policy: token.actions.githubusercontent.com principal
  • Condition: sub: repo:org/repo:ref:refs/heads/main
  • Workflow uses aws-actions/configure-aws-credentials
  • Zero AWS secrets stored in GitHub — token exchange only
🪣 S3 Bucket Policies
  • Block all public access on all dbt-related buckets
  • Enable default encryption (SSE-S3 or SSE-KMS)
  • Bucket policy restricts access to specific IAM role ARNs only
  • Lifecycle rules auto-expire old Athena query results (7 days)
  • Enable versioning on the dbt artifacts bucket for rollback
🔍 Athena Workgroup Controls
  • Enforce workgroup-level results location (override user settings)
  • Set per-query data scan limit (e.g. 1 TB max) to prevent runaway costs
  • Enable CloudWatch metrics per workgroup for cost attribution
  • Separate workgroups for prod, dev, and BI dashboards
  • IAM condition keys scope roles to specific workgroups only
🔍 Monitoring & Audit
  • AWS CloudTrail — full API audit log for Athena, S3, Glue, IAM
  • Athena query history — per-query execution time, bytes scanned
  • CloudWatch Logs — dbt stdout/stderr for every run
  • AWS Cost Explorer — tag dbt runs with Project=dbt for cost visibility
  • IAM Access Analyzer — detect over-permissioned roles automatically

Cost Optimization

Athena charges $5 per TB scanned — design to minimize
📦
Use Columnar Formats
Store all dbt outputs as Parquet with Snappy compression. Athena only reads the columns your query touches. A query on 2 columns out of 50 reads ~4% of the data. ORC is also excellent. Avoid CSV/JSON in production marts.
📂
Partition Strategically
Use partitioned_by = ['event_date'] in dbt model configs. Athena uses partition pruning to skip entire S3 prefixes. A query for a single day on a 3-year dataset reads ~0.1% of the data. Balance partition granularity — too many small files hurts.
📏
Right-size File Counts
Athena performs best with files between 128MB–1GB each. Too many tiny files (the "small files problem") causes slow metadata operations. Use the optimize_write config in dbt-athena or schedule periodic OPTIMIZE / vacuum jobs.
